Core triage guided workflow
General incident triage

Start broad when the symptom is real but the failure domain is still fuzzy.

Typical symptoms
  • Customer impact reported but service owner is unsure where the fault started.
  • Multiple alarms or dashboards look noisy and you need the shortest path to a likely failure domain.
OpenShift evidence sources
  • Cluster operators and warning events
  • Recent rollout, upgrade, or configuration changes
  • Top-level dependencies across nodes, workloads, routes, storage, and operator lifecycle
Investigation steps
  • Confirm the first bad timestamp and blast radius.
  • Compare healthy versus unhealthy nodes, namespaces, routes, or worker pools.
  • List the next safe validation steps if the first hypothesis is wrong.
Generated investigation prompt

Act as an OpenShift incident commander for read-only troubleshooting. Investigate the operator issue by gathering evidence from cluster operators, warning events, rollout changes, node state, and major upstream or downstream dependencies. Return: 1) likely root cause, 2) evidence collected, 3) blast radius, 4) immediate next safe checks, and 5) deeper follow-up checks if the first hypothesis is wrong.

  • symptom-first incident triage
  • cluster operators, events, routes, workload, and topology checks
  • blast radius and dependency mapping
  • next steps for operator execution

Choose between the local Ollama runtime and supported external providers such as OpenAI, Azure OpenAI, Anthropic, Gemini, and OpenRouter.

Tip: the guided posture module is ideal for cluster reviews, while the troubleshooting module is better for incident response and platform diagnosis.
